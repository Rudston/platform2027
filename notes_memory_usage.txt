Good instinct — this is exactly the kind of thing that's cheap to get right now (patterns) and expensive to retrofit later (rewriting queries across a live app). The short version: yes, there are clearly two zones, and you've actually already drawn the line correctly in a few places without naming it as a principle. Let me make the principle explicit so it stays consistent as new features get added.
The two zones
Zone A — "My stuff": small, bounded, eager-load fully.
A user's own circle memberships. Even a very active member is realistically in dozens of circles, not thousands. This is the dashboard case you're describing: on login/dashboard-load, eager-load $user->activeMemberships()->with(['circle.circleable', 'circle.locatable']) — hydrate the whole set, once, per request. There's no pagination-vs-lazy question here because the ceiling is small by construction (nobody joins 5,000 circles — canUserJoin() rules already cap concurrent memberships per type anyway).
Zone B — "The tree": unbounded, always query one level/window at a time.
Anything touching the wider Circle tree (Explore's column browser, search, "14 Campaigns in sub-regions" counts) must never eager-load the tree — it must stay query-scoped to exactly what's rendered. And you're already doing this correctly:

The column browser only ever fetches direct children of the currently-selected node (not the whole subtree) — that's the natural "on demand" pattern already.
The empty-state descendant counts use path LIKE for a count, not a hydrated collection — good, because a count query doesn't care how many rows match, it never materializes them.
Search caps at 10 results with a 2-character minimum before firing.

So the rule of thumb: eager-load when the result set size is bounded by "how many things belong to this user," lazy/windowed-load when the result set size is bounded by "how big is South Africa's admin hierarchy." The tree is the thing that can have thousands of nodes; a person's slice of it never is.
Where this will bite if not planned for
A few places worth flagging now, before the dashboard gets built, since they're the ones most likely to accidentally cross from Zone A into Zone B:

Member/participant counts on a list of "my circles." If the dashboard shows 6 circle cards each with a member count, don't do 6 separate count queries — same pattern as ExploreCommunities::memberCounts() (one grouped query keyed by circle_id, passed down). You already have the idiom; just needs to be reapplied here.
"Recent activity across my circles" (forum discussions, if/when you want a dashboard feed of "new posts in your communities") — this is the one to be careful with. It's tempting to write Circle::whereIn('id', $myCircleIds)->with('forumDiscussions'), but if a circle has thousands of discussions accumulated over time, that's an unbounded eager load per circle. This wants latest()->limit(N) per circle (or a single query across all the user's circle_ids ordered by created_at with an overall limit, then group in PHP) — bounded by "how much do I show," not "how much exists."
Ancestor breadcrumbs. Building a breadcrumb from a MainPlace back up to Country is naturally bounded (max 5 levels), so this one's fine to eager-load regardless of tree size — worth naming explicitly so nobody "optimizes" it into a lazy/paginated thing it doesn't need to be.
Circle administrators / responsibleAdminFor's ancestor walk. Also naturally bounded by tree depth (≤5), not tree width, so eager-loading up the ancestor chain is fine and cheap. The danger zone is always sideways (siblings/descendants), not upward (ancestors), because siblings/descendants are the axis with no ceiling.
Tags on a circle/forum group list. <x-tag-list> per card — if a dashboard shows 6 circles each with tags, batch-load via a single whereIn on the polymorphic pivot rather than N+1 per card, same shape as memberCircleIds().

The general query-cost heuristic
A useful gut-check for any new dashboard/list feature going forward: "is the thing I'm iterating over bounded by the user, or bounded by the platform?" If bounded by the user (their memberships, their created content, their notifications) — eager-load, don't overthink it, it'll never be large. If bounded by the platform (all circles at a level, all discussions in a group, all descendants of a node) — always query with an explicit limit/pagination/level-scoping, never ->get() the whole thing "because it'll probably be fine," because on a civic platform meant to scale to real national usage, it eventually won't be.
One more thing worth deciding once, not per-feature: whether dashboard data gets cached (even a short TTL, like ContentBlock/EmailTemplate already do) since "my 6 circles + their member counts" changes rarely relative to how often a user loads their dashboard. That's a smaller decision than the eager/lazy split above, but it's the natural next lever once the query patterns are already correct.
