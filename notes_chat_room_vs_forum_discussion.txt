On a related topic. With our developing 'comments' / 'posts' etc structures currently only used for forum discussions, once proper 'polling'
is implemented how would the requirements differ for a 'chat room'??


Answer:

Genuinely different problem in a few ways, even though the underlying Comment/Like tables are generic enough to technically serve both. Let me split it into: what carries over unchanged, what needs adapting, and what's a new requirement that doesn't exist in the forum model at all.
What carries over as-is

The generic commentable_type/commentable_id polymorphism — a chat room could be a commentable type exactly like ForumDiscussion, no schema change needed there.
Like — reactions on a chat message, same table.
Author/edit/delete mechanics — a chat message being editable, deletable, tombstoned-if-replied-to (if you keep threading at all) can reuse what's already built.
The moderation queue concept (comment_moderation_records) — still useful as a record of "this got flagged," regardless of channel.

What needs to change shape
Latency — this is the big one. Forum discussions tolerate wire:poll.10s because reading a forum thread is inherently asynchronous; nobody expects a reply to appear the instant it's posted. A chat room is the opposite — polling every 10 seconds would feel broken, not "a bit slow." This is exactly the boundary where Tier 0 (polling) stops being adequate and Tier 1 (Reverb + Echo + broadcast events + channel auth) becomes non-optional rather than a nice-to-have. Chat is the natural first real use case that forces that infrastructure decision, whereas forums can coast on polling indefinitely.
Structure — flat stream vs. nested tree. Forum replies are a genuine tree (reply-to-a-reply-to-a-reply, 3-level indent cap, etc.). Chat is almost always a flat, append-only sequence — parent_id either doesn't exist at all, or exists only as a lightweight "quoting/replying to X" reference for context, not a rendering hierarchy. Trying to render a live chat room with real nested-thread indentation gets unreadable fast at chat speed; that's why Slack/Discord-style UIs show quote-replies inline rather than indenting.
Query pattern — keyset pagination vs. small full loads. A forum discussion's comment count is modest and gets loaded close to fully. A chat room accumulates messages continuously and forever — you need cursor/keyset pagination ("give me the last 50, then load older on scroll-up"), not ->get(). This is the same eager/lazy principle from your dashboard question earlier, just showing up in a new place: a chat room's history is Zone B (unbounded, windowed), even though "the last 50 messages" that render on open is Zone A-like (bounded, always fetched).
Moderation timing — this one's a genuine problem, not just an adaptation. The pending-review quarantine pattern just built works because forum comments are fetched, not pushed — a flagged comment can sit invisible to everyone but its author until an admin acts, and nobody's missed anything because nobody was watching it live. In a broadcast chat room, the message has already been pushed to every connected client's screen the instant it was sent — by the time an AI check (even a fast one) comes back, everyone in the room may have already seen it. You can't quarantine something that's already been shown. This forces a real choice for chat that forums don't need to make: either (a) synchronous pre-send moderation check (adds latency to every single message — the tradeoff you just avoided for forums by making it reactive/async), or (b) accept a "flash then retract" pattern (bad message briefly visible, then removed/replaced with a tombstone once flagged) and treat that brief exposure as an accepted cost. Worth deciding deliberately whenever chat becomes real, not something to solve today.
Genuinely new requirements chat has and forums don't

Presence — who's currently in the room, typing indicators. This needs a presence channel (a different Echo channel type than the private channel a forum discussion would use), and it's inherently ephemeral state — it doesn't belong in a persisted table the way comments do.
Read state / unread counts — "you have 4 unread messages in this room" is a per-user last_read_at concept, structurally different from forum_discussion_participants' join/leave semantics (which is about permission to participate, not what have I seen).
Rate limiting — spam throttling per user per room matters far more for chat (messages-per-second) than for forum posting, where the natural friction of writing a longer post already self-limits volume.

So the short version: the data layer mostly transfers, but latency, moderation timing, and read/presence state don't — and moderation timing in particular is a real architectural fork, not just a config change, whenever you actually get there. Nothing here needs deciding today since chat rooms aren't in scope yet — just flagging it so it's not a surprise when that feature eventually gets designed.
